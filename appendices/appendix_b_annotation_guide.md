# Supplement B — Annotation Guide for Gold-Standard Benchmark Construction

**Study:** Auditable Concept Harmonisation in Bibliometric Analysis: Benchmarking an LLM-DAG Workflow
**Document version:** 1.1 (revised)
**For revision after pilot calibration meeting**

---

## Section 1: Purpose and Scope

This guide supports the annotation of 500 keyword pairs drawn from a circular economy Scopus corpus (2017–2024, 26,535 records). These pairs form the gold-standard benchmark used to evaluate an LLM-DAG concept harmonisation pipeline and six baseline methods.

**Your task:** For each pair, decide whether the two keyword strings represent the **same bibliometric concept** (match), **different concepts** (non-match), or whether the available evidence is **insufficient for a safe decision** (uncertain).

The three-label scheme is deliberately conservative. A *match* decision authorises merging two keyword strings into a single concept node in a bibliometric thematic map — a consequential act that permanently collapses two pieces of evidence into one. A *non-match* decision preserves the distinction. An *uncertain* decision defers to human adjudication rather than risking a false merge.

**The governing principle is: when in doubt, label uncertain.** A false positive (merging distinct concepts) is typically harder to detect and reverse downstream than a false negative, because an incorrect merge propagates silently through all subsequent analysis.

## Section 2: Scope Policy (Table 1)

For every pair, identify which case type applies and apply the default label unless you have specific grounds to deviate. If you deviate, explain in your annotation notes.

| Case type | Default label | Rationale | Example pair |
|---|---|---|---|
| Spelling variant | Match | Orthographic variation does not change the represented concept | behaviour analysis / behavior analysis |
| Punctuation or hyphenation variant | Match | Formatting difference only; the concept is the same | machine-learning / machine learning |
| Singular or plural variant | Match | Number change usually preserves concept identity in bibliometric contexts | neural network / neural networks |
| Acronym and expanded form | Match if unambiguous | Merge only when the acronym clearly denotes the same concept; if polysemous, label uncertain | MSW / municipal solid waste |
| Multilingual or transliterated equivalent | Match if equivalence is clear; otherwise uncertain | Translation alone is insufficient unless same-concept meaning is defensible | inteligencia artificial / artificial intelligence |
| Near-synonym with clear same-concept use | Match | Merge only when the pair functions as one concept unit under scope; otherwise non-match | COVID-19 / coronavirus disease 2019 |
| Broader–narrower relation | Non-match | Different conceptual granularity should not be collapsed | recycling / plastic recycling |
| Related but distinct concepts | Non-match | Semantic proximity is not conceptual identity | bibliometrics / scientometrics |
| Ambiguous acronym or short form | Uncertain | One or both strings may denote multiple concepts; do not resolve by guessing | IR / information retrieval |
| Context-dependent overlap | Uncertain | Even experts may genuinely disagree without additional context | digitalisation / digital transformation |
| Malformed, truncated, or underspecified string | Uncertain | Safe adjudication is not possible from the string pair alone | bibliomet* / bibliometric analysis |

## Section 3: Worked Examples — One Per Difficulty Stratum

The following examples are drawn directly from the actual `candidate_pairs.csv` benchmark file. Every pair you see during annotation was generated by the same process and comes from the same corpus.

---

### Stratum i — Capitalisation / Whitespace Variants

**Pair:** `Compost quality` / `compost quality`
**Corpus frequencies:** 2 / 2
**Retrieval method:** capitalisation_whitespace
**Assigned label: MATCH**

*Justification:* The two strings differ only in the capitalisation of the first letter. Capitalisation alone does not distinguish concepts in bibliometric keyword strings; it reflects authorial inconsistency in how keywords were entered at submission.

---

### Stratum ii — Spelling Variants (Jaro–Winkler Similarity [0.85, 0.95))

**Pair:** `Circular finance` / `Circular waste management`
**Corpus frequencies:** 3 / 2
**Similarity score:** 0.851
**Assigned label: NON-MATCH**

*Justification:* This pair was retrieved by Jaro–Winkler fuzzy blocking because both strings begin with "Circular". However, "circular finance" denotes financial instruments supporting the circular economy transition, while "circular waste management" denotes practices for managing waste flows. These are distinct research streams with different conceptual referents. The shared prefix "Circular" creates surface similarity but does not imply conceptual identity.

---

### Stratum iii — Acronym and Expanded Form

**Pair:** `MSW` / `municipal solid waste (MSW)`
**Corpus frequencies:** 13 / 9
**Retrieval method:** acronym_expansion (pair ID: BP0118)
**Assigned label: MATCH**

*Justification:* The long-form keyword contains the acronym in parentheses as a parenthetical self-label: "municipal solid waste (MSW)". This is unambiguous evidence that the keyword author intended "MSW" to denote exactly this concept. "MSW" is a standard, domain-stable abbreviation in circular economy research with no other common expansion in this literature.

*Boundary case note:* If an acronym had multiple plausible expansions in this corpus (e.g., "CE" = circular economy, carbon emissions, or chemical engineering), label uncertain unless the partner string fully disambiguates it.

---

### Stratum iv — Punctuation / Hyphenation Variants

**Pair:** `By-products` / `byproducts`
**Corpus frequencies:** 52 / 12
**Retrieval method:** punctuation_hyphenation
**Assigned label: MATCH**

*Justification:* "By-products" and "byproducts" are two standard orthographic conventions for the same English word. Hyphenation conventions vary by style guide, journal, and author nationality. The frequency asymmetry (52 vs. 12) reflects spelling convention preferences in this corpus, not a conceptual distinction.

---

### Stratum v — Singular / Plural Variants

**Pair:** `Biogas plant` / `biogas plants`
**Corpus frequencies:** 15 / 3
**Retrieval method:** singular_plural_s
**Assigned label: MATCH**

*Justification:* "Biogas plant" and "biogas plants" differ only in grammatical number. In bibliometric keyword analysis, singular and plural forms of the same noun phrase are conventionally treated as identical concepts. The lower frequency of "biogas plants" (3 vs. 15) reflects a minority author preference, not a distinct sub-concept.

*Boundary case note:* Some singular/plural pairs mark genuine distinctions (e.g., "value" vs. "values"), but this is rare for multi-word phrases.

---

### Stratum vi — Near-Synonyms (Embedding Cosine Similarity [0.75, 0.85))

**Pair:** `Biodegradable packaging` / `sustainable packaging`
**Corpus frequencies:** 5 / 11
**Similarity score:** 0.766
**Retrieval method:** embedding_near_synonym
**Assigned label: NON-MATCH**

*Justification:* These strings are semantically proximate but not conceptually identical. "Biodegradable packaging" denotes a specific material property: the ability to decompose biologically. "Sustainable packaging" is a broader concept encompassing biodegradability, recyclability, reduced material use, and lifecycle considerations. Collapsing these would silently merge a specific material technology topic with a broader sustainability research stream, losing meaningful analytical granularity.

*Boundary case note:* Consult auxiliary context (titles) only if genuinely unsure whether authors in this corpus treat the terms as interchangeable.

---

### Stratum vii — Broader–Narrower Proxies (Embedding Cosine [0.60, 0.75))

**Pair:** `Additive Manufacturing` / `digital fabrication`
**Corpus frequencies:** 3 / 3
**Similarity score:** 0.698
**Retrieval method:** embedding_broader_narrower
**Assigned label: NON-MATCH**

*Justification:* "Additive manufacturing" refers to a specific family of production technologies (3D printing, layer-by-layer deposition). "Digital fabrication" is broader, encompassing additive manufacturing, subtractive manufacturing (CNC milling), robotic assembly, and other computer-controlled methods. Additive manufacturing is a proper subset of digital fabrication — not its equivalent.

*Boundary case note:* This stratum produces many non-matches by design. Annotators should expect a higher non-match rate here than in strata i–v.

---

### Stratum viii — Ambiguous Short Forms (2–4 character strings + embedding)

**Pair:** `3Rs` / `4R`
**Corpus frequencies:** 4 / 2
**Similarity score:** 0.766
**Retrieval method:** embedding_short_ambig
**Assigned label: UNCERTAIN**

*Justification:* "3Rs" in the circular economy literature typically refers to "Reduce, Reuse, Recycle". "4R" may refer to "Reduce, Reuse, Recycle, Recover", or other four-principle formulations depending on the author's framework. Whether these should be merged depends on whether corpus authors treated them as synonyms or as distinct frameworks. Without consulting auxiliary context, the decision is genuinely uncertain; even with context, annotators may reasonably disagree.

*Boundary case note:* Short-form ambiguity is high in this stratum — expect frequent uncertain labels.

---

### Stratum ix — Malformed / Encoding-Artefact Strings

**Pair:** `Fe<sup>3+</sup> cation` / `Fe-TiO<sub>2</sub> composite`
**Corpus frequencies:** 1 / 2
**Similarity score:** 0.788
**Retrieval method:** malformed_jw_nearest (pair ID: BP0194)
**Assigned label: UNCERTAIN**

*Justification:* Both strings contain unresolved HTML rendering tags (`<sup>` and `<sub>`) introduced during Scopus CSV export. The intended strings are `Fe³⁺ cation` (iron(III) ion) and `Fe-TiO₂ composite` (iron–titanium dioxide material). Even after interpreting the tags, the strings denote different chemical entities. However, the correct label is **uncertain**, not non-match: when a string is malformed, the annotator cannot be certain what the author originally intended, and safe adjudication is not possible from the malformed string pair alone.

*Boundary case note:* The malformation itself — not the inferred chemistry — is what makes the label uncertain. If the strings were correctly encoded, the decision would be non-match.

---

### Stratum x — Weak Semantic Links (Embedding Cosine [0.50, 0.60))

**Pair:** `2030 Agenda` / `agenda setting`
**Corpus frequencies:** 6 / 2
**Similarity score:** 0.571
**Retrieval method:** embedding_weak_semantic
**Assigned label: NON-MATCH**

*Justification:* "2030 Agenda" refers to the UN Sustainable Development Goals framework (adopted 2015). "Agenda setting" is a concept from political science and communication theory about how media or policymakers influence public priorities. These share the word "agenda" but denote entirely different concepts from different research traditions. Annotators should expect a high non-match rate in stratum x — these are deliberately weak signals included to test the model's ability to reject near-misses.

## Section 4: Decision Rules for Boundary Cases

### Rule 1: Acronym Polysemy

If an acronym has more than one common expansion in the circular economy literature, label **uncertain** UNLESS the paired keyword disambiguates it completely (e.g., the expansion appears in parentheses within the longer string).

Examples of polysemous acronyms in this corpus: CE (Circular Economy, Carbon Emissions, Consumer Electronics, Chemical Engineering), LCA (Life Cycle Assessment, Latent Class Analysis), MFA (Material Flow Analysis, Multi-Factor Authentication).

**Test:** Could this acronym plausibly refer to more than one concept in a circular economy corpus? If yes → uncertain (unless the partner string disambiguates).

Parentheses are not required for disambiguation. A partner string may be sufficient when it is a clear and unique lexical expansion of the acronym in this corpus; if reasonable ambiguity remains, label uncertain unless auxiliary titles resolve it.

### Rule 2: Near-Synonym vs. Related-but-Distinct

The diagnostic question: **"In a co-word thematic map of this corpus, should these two strings be merged into ONE concept node?"**

- If keeping them separate creates *artificial fragmentation* of one research concept → **match**
- If merging them *loses meaningful analytical distinction* between two research streams → **non-match**

The question is whether merging *preserves* or *distorts* the thematic structure of the literature.

### Rule 3: Broader–Narrower Relations

If one keyword is a clear subcategory of the other, label **non-match**. Conceptual containment is not conceptual identity.

- "Recycling" / "plastic recycling" → non-match (plastic recycling is a specific instance of recycling)
- "Renewable energy" / "solar energy" → non-match (solar is a specific type of renewable)
- "Life cycle assessment" / "consequential LCA" → non-match (a specific methodological variant)

**Test:** Can you construct "All X is Y, but not all Y is X"? If yes → non-match.

### Rule 4: Partial Lexical Overlap

Two keywords may share a word but denote entirely different concepts. Ask: is the shared word doing conceptual work (narrowing meaning in the same direction) or is it incidental (same modifier applied to different base concepts)?

- "Green supply chain" / "green economy" → non-match (supply chain management vs. macroeconomic framework)
- "Digital transformation" / "digital twin" → non-match (organisational change vs. simulation technology)

### Rule 5: Frequency Asymmetry

**Frequency is irrelevant to the equivalence decision.** A singleton keyword paired with a high-frequency keyword is not automatically a non-match. Use frequency only as a heuristic sanity check for potential typos or encoding artefacts.

### Rule 6: When to Consult Auxiliary Context

If keyword strings are genuinely ambiguous after applying the scope policy, consult up to **three representative titles** per keyword from the corpus. Use titles only as disambiguation evidence — to determine what concept each keyword denotes — not to assess whether the papers are topically similar. Note in your annotation: "Consulted auxiliary context: [title 1], [title 2]."

### Rule 7: When to Label Uncertain

Label **uncertain** when:

(a) After consulting titles, you still cannot confidently decide.
(b) The acronym is polysemous and the pair does not disambiguate it.
(c) One or both strings are malformed, truncated, or contain encoding artefacts.
(d) A reasonable, well-calibrated colleague might genuinely disagree because the evidence is insufficient.

Uncertain is the correct label when evidence does not support a confident binary decision. The adjudicator resolves all uncertain and disagreement pairs by applying the same scope policy and auxiliary-context rules as the primary annotators.

For pairs consisting only of short or opaque acronyms without an in-pair expansion (e.g., `3DP` / `PPP`), prioritise uncertain over non-match; do not use domain knowledge alone to resolve acronyms that the strings themselves do not define.

## Section 5: Annotation Procedure

> **Important:** The retrieval stratum indicates why a pair was proposed as a candidate (e.g., high embedding similarity, fuzzy string match), but it does not imply the expected label. A pair from the "near-synonym" stratum may still be a non-match; a pair from the "broader–narrower" stratum may turn out to be a match. Apply the scope policy to each pair independently, regardless of its assigned stratum.

Follow these steps for every pair:

1. Read `keyword_a` and `keyword_b`. Check the `stratum` column to orient yourself to the difficulty type and retrieval method.

2. Apply the scope policy (Section 2). If the case type is clear, apply the default label.

3. If ambiguous, work through the boundary case rules (Section 4).

4. Enter your label in the `annotator1_label` column (or `annotator2_label`). Valid labels: **match**, **non-match**, **uncertain**.

5. Enter a justification in the notes column. **Required for every pair.** Write 1–2 sentences explaining your decision. If you consulted auxiliary context, note which titles you read.

6. Do NOT consult the other annotator's spreadsheet, any system outputs (LLM pipeline, baseline outputs), or published keyword maps.

   Annotators may pause and resume across sessions. If a pair is difficult, flag it and return to it within the same annotation round. Annotators must not revise earlier labels after seeing the other annotator's decisions.

7. Save your file after each session using the naming convention in `ANNOTATION_README.md`.

### Label format note

In the annotation spreadsheet, use `match`, `non-match`, and `uncertain` (hyphen in non-match). In the pipeline's JSON output schema (Supplement A), the corresponding value is `non_match` (underscore). These are mapped to the same class during evaluation.

## Section 6: Pilot Round Instructions

### Purpose

The pilot round (1) calibrates annotator understanding of the scope policy before labelling the full 500 pairs; (2) identifies ambiguous rules in this guide; and (3) establishes a shared understanding of boundary cases before independent labelling.

### Procedure

1. Both annotators receive their personal pilot spreadsheet (`pilot_annotator1.csv` or `pilot_annotator2.csv`) containing 50 pairs sampled proportionally from all 10 strata.
2. Read this guide in full before beginning. Do not start labelling until Sections 1–5 are read.
3. Label all 50 pilot pairs **independently**. Do not discuss specific pairs before both sheets are complete.
4. Save as `pilot_annotator1_COMPLETED.csv` and `pilot_annotator2_COMPLETED.csv`. Submit to the adjudicator.

### Calibration Meeting

The adjudicator runs `compute_agreement.py` on both completed pilot files. Both annotators and adjudicator then review all disagreement pairs, determine which label is correct under the scope policy, identify rule ambiguities, and revise this guide accordingly.

### Re-annotation

After the calibration meeting, both annotators **re-label all 50 pilot pairs** under the revised guide. Pre-calibration labels are discarded. Re-labelled pairs carry the same weight as any other pair in the final benchmark.

### Expected Pilot Agreement

Target: Cohen's kappa κ ≥ 0.70. If κ < 0.60, additional calibration rounds may be needed before the full annotation round.
